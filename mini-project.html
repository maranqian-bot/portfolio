MapingoAI 영어 회화 코칭 플랫폼

Mapingo는 사용자가 지도에서 고른 실제 장소와 이전 학습 기록을 바탕으로 더 맥락적인 영어 회화 연습을 제공하는 AI 학습 서비스입니다.

Overview

담당 역할 · 설계 중심

담당 역할

  • AI Coaching 기능 담당 및 FastAPI 기반 AI 서버 구현
  • Azure Speech · OpenAI 연동과 발음 평가 · 최종 피드백 흐름 구현
  • React · Spring · FastAPI 사이 음성 업로드와 응답 계약 안정화
  • 스크립트 생성 → 음성 녹음 → 발음 평가 → 최종 피드백 → 후속 추천까지 하나의 학습 흐름 완성

단순히 AI 응답을 보여주는 데서 끝나지 않고, 사용자가 실제로 끝까지 학습 결과를 받을 수 있도록 안정적인 코칭 흐름을 설계하고 구현했습니다.

설계 중심

Mapingo의 AI Coaching은 OpenAI, Azure Speech, Spring API, React UI가 하나의 학습 흐름으로 이어져야 했기 때문에, 어느 한 단계만 흔들려도 사용자 경험 전체가 쉽게 끊길 수 있었습니다. 그래서 저는 AI 응답 품질만 높이는 데서 그치지 않고, LLM 응답 계약, 브라우저별 음성 포맷 차이, multipart 업로드 경계, 외부 API timeout과 fallback까지 함께 설계했습니다. 결국 이 프로젝트에서의 설계 중심은 “좋은 AI 결과를 만든다”보다 “스크립트 생성부터 최종 피드백과 후속 추천까지 사용자가 끝까지 도달할 수 있게 만든다”에 더 가까웠습니다.

형식 자유도 vs 서비스 안정성

Prompt 품질보다 응답 계약을 먼저 고정

LLM은 내용이 좋아도 응답 형식이 조금만 흔들리면 Spring과 React에서 그대로 파싱 오류로 이어질 수 있었습니다. 그래서 먼저 output format과 schema를 강하게 고정하고, 그 안에서 script와 feedback 품질을 높이는 방향으로 설계했습니다.

이후 problemWords normalize 같은 후처리를 추가해 문자열 배열과 객체 배열이 섞여도 서비스 응답이 깨지지 않도록 보완했습니다.

“더 자연스러운 답변”보다 “서비스가 끝까지 소비할 수 있는 답변”을 우선한 결정이었습니다.

클라이언트 분기 vs 서버 표준화

브라우저 음성 입력은 서버에서 표준 포맷으로 정규화

브라우저마다 MediaRecorder 포맷과 codec이 달라 프론트에서 모든 경우를 직접 처리하면 복잡도가 크게 올라갔습니다. 그래서 webm 업로드는 FastAPI에서 ffmpeg 기반 wav로 변환해 Azure Speech가 안정적으로 받을 수 있는 입력으로 통일했습니다.

여기에 pronunciation assessment 실패 시 STT fallback을 붙여, 포맷 차이나 인식 실패가 있어도 전체 코칭 흐름이 끊기지 않게 했습니다.

입력 다양성을 프론트에서 모두 흡수하기보다, 서버 쪽에서 표준화와 fallback을 함께 두는 편이 더 운영 가능하다고 판단했습니다.

Troubleshooting

트러블슈팅

Mapingo에서는 AI 품질 문제보다도, 응답 형식과 음성 업로드처럼 경계면에서 생기는 문제가 서비스 전체를 더 쉽게 흔들었습니다. 아래 두 사례가 제가 실제로 해결한 대표적인 문제입니다.

LLM 응답 형식이 바뀌면 파싱 실패하던 문제

AI 회화 스크립트와 최종 피드백은 화면에서 바로 렌더링해야 했기 때문에, 응답 구조가 조금만 흔들려도 Spring DTO 역직렬화와 프론트 표시가 함께 깨질 수 있었습니다. 그래서 프롬프트를 Role · Instruction · Context · Output Format으로 더 강하게 고정하고, problemWords normalize 후처리를 넣어 문자열 배열과 객체 배열이 섞여도 서비스가 끝까지 응답을 소비할 수 있게 정리했습니다.이 경험을 통해 AI 응답은 “좋은 문장”보다 “서비스가 안전하게 파싱할 수 있는 구조”가 먼저 확보되어야 한다는 점을 배웠습니다.

React–Spring–FastAPI 음성 연동 오류

React에서 녹음은 정상적으로 끝나도, 마지막 chunk가 반영되기 전에 파일이 만들어지거나 Spring에서 multipart content type이 바뀌면 FastAPI 쪽 음성 처리가 쉽게 실패했습니다. 이를 해결하기 위해 React에서는 녹음 종료 후 최종 File이 완성된 뒤 업로드하도록 정리했고, Spring에서는 원본 content type을 보존해 FastAPI가 실제 음성 형식을 잃지 않도록 바꿨습니다. 동시에 FastAPI 응답도 음성 평가 실패 시 기본 구조를 유지하도록 정리해 세 서버 간 계약을 안정화했습니다.결국 문제는 한 서버 안이 아니라 경계면 여러 곳에 나뉘어 있었고, 전체 요청 흐름으로 봐야만 원인을 정확히 잡을 수 있다는 점을 배웠습니다.

협업

  • AI 코칭 기능을 진입 화면, 세션 생성, 대화 진행, 결과·추천 흐름으로 나눠 역할을 먼저 맞추고, 각 단계에서 어떤 서버가 어떤 책임을 가지는지 문서와 PR 설명에 함께 남겼습니다.
  • FastAPI 응답 필드, Spring DTO, React 화면 소비 구조가 어긋나지 않도록 PR 단위로 계약을 같이 맞췄고, 바뀐 응답 형식은 프론트와 백엔드가 동시에 확인할 수 있게 공유했습니다.
  • 음성 업로드 오류가 났을 때는 React Network 요청, Spring multipart 로그, FastAPI 진단 로그를 한 흐름으로 묶어 비교하면서 문제를 좁혀 갔습니다.
  • 배포와 재현이 막히지 않도록 GitHub Actions, 환경변수, fallback 로그 같은 운영 요소도 기능 구현과 분리하지 않고 함께 정리했습니다.

이 프로젝트에서 배운 점

Mapingo를 하면서 AI 기능은 단순히 모델을 붙이는 일이 아니라, 입력 포맷, 응답 계약, 실패 시 fallback, 사용자에게 보여줄 안내까지 함께 설계해야 비로소 서비스가 된다는 점을 배웠습니다.

특히 음성 기반 기능은 한 서버 안에서 끝나는 문제가 거의 없었습니다. 브라우저 녹음, Spring 전달, FastAPI 처리, Azure Speech와 OpenAI 응답까지 모두 이어진 경계 문제로 봐야 원인을 정확히 찾을 수 있었습니다.

그리고 작은 로그 한 줄, 기본 응답 구조 하나를 미리 보장해 두는 것만으로도 팀 전체 디버깅 속도가 크게 달라진다는 점을 실제로 체감했습니다.
