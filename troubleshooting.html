CareerWave

JWT 인증 필터 예외가 500으로 반환되는 문제

PR #351
Problem

만료되었거나 위변조된 토큰으로 보호된 API를 호출했을 때 클라이언트가 기대한 401 응답이 아니라 500 응답이 반환되었습니다. 프론트엔드에서는 로그인 만료와 서버 오류를 구분할 수 없어 재로그인 안내 흐름이 깨졌습니다.

Cause

처음에는 전역 예외 처리 설정 문제로 보고 Controller Advice를 확인했지만, JWT 검증은 DispatcherServlet에 도달하기 전 Security Filter Chain에서 실행되고 있었습니다. 따라서 필터 내부에서 발생한 토큰 만료, 서명 불일치 예외는 Controller Advice가 잡을 수 없었고, 별도 처리 없이 RuntimeException처럼 흘러가며 500으로 변환되었습니다.

Solution

JWT 필터에서는 토큰 검증 실패 원인을 AuthenticationException 계열로 정리하고, AuthenticationEntryPoint에서 401 상태 코드와 에러 메시지를 명시적으로 반환하도록 처리했습니다. 케이스별 메시지가 과하게 노출되지 않도록 클라이언트 안내용 메시지와 서버 로그를 분리했습니다.

Result

토큰 누락, 만료, 위변조 요청을 각각 재현해 모두 401로 응답되는지 확인했습니다. 정상 토큰 요청은 기존 API 흐름을 유지했고, 프론트엔드는 401을 기준으로 재로그인 안내를 처리할 수 있게 되었습니다.

새로고침 시 동시 refresh 경쟁으로 로그아웃

PR #971
Problem

페이지 새로고침이나 OAuth 콜백 직후처럼 access token이 아직 메모리에 복구되지 않은 순간, 여러 컴포넌트가 동시에 세션 복원을 시도하면서 사용자가 바로 로그아웃되는 문제가 있었습니다. 정상적인 refresh 쿠키가 있어도 화면 진입 타이밍에 따라 세션이 끊겨 UX가 불안정했습니다.

Cause

401 응답이나 초기 보호 라우트 진입 시 각 요청이 독립적으로 /token/refresh를 호출하고 있었고, 서버는 refresh token rotation과 재사용 탐지를 적용하고 있었습니다. 같은 쿠키로 거의 동시에 두 번 이상 재발급이 들어오면 먼저 회전된 토큰이 뒤늦게 다시 사용된 것으로 판단되어 세션 전체가 폐기될 수 있었습니다.

Solution

프론트엔드의 세션 복원 로직을 single-flight로 묶어 진행 중인 refresh Promise에 후속 요청이 합류하도록 바꿨습니다. 또한 OAuth 콜백 페이지에서 probeAuth()로 refresh 쿠키 회전이 끝나기 전에 이동하지 않도록 순서를 고정해, 새 페이지가 구 쿠키로 다시 refresh를 발사하는 경쟁 창도 닫았습니다.

Result

probeAuth를 동시에 여러 번 호출해도 refresh 요청이 1회만 나가는 테스트로 재현 케이스를 고정했습니다. OAuth 콜백 이후에도 회전 완료 전에는 이동하지 않도록 검증해, 새로고침 직후 즉시 로그아웃되던 현상이 사라졌습니다.

미완성 계정에도 일반 토큰이 발급될 위험

PR #488
Problem

소셜 로그인 제공자 인증은 성공했지만 서비스 회원가입이 아직 끝나지 않은 계정에도 일반 access/refresh token이 발급될 수 있는 경계가 있었습니다. 이 경우 약관 동의나 추가 정보 입력 전에 보호 API에 접근할 수 있어 가입 단계 분리가 무너질 수 있었습니다.

Cause

초기 설계에서는 OAuth callback 성공과 서비스 회원 생성 완료를 같은 인증 성공으로 취급하기 쉬웠습니다. 또한 state 검증과 가입 완료 신호가 분리되어 있지 않으면, callback 직후 미완성 계정을 완전한 로그인 세션처럼 다루게 됩니다.

Solution

Provider 인증 완료를 나타내는 Redis state와, 서비스 가입 완료 전까지만 쓰는 10분 만료·1회성 Social Signup Token을 분리했습니다. Social Signup Token 저장소는 getAndDelete() 원자 연산으로 소비되도록 구현해 중복 사용을 막고, 일반 JWT는 가입이 완료된 뒤에만 발급되도록 흐름을 나눴습니다.

Result

OAuth callback 이후 미완성 계정은 일반 세션 대신 가입 마무리용 토큰만 받도록 경계를 분리했습니다. state 불일치와 중복 소비 테스트를 추가해, provider 인증 완료와 서비스 가입 완료가 서로 다른 단계로 유지되는 것을 확인했습니다.

로그아웃 후에도 JWT가 남아 있으면 세션을 즉시 끊을 수 없는 문제

PR #361
Problem

초기 JWT 인증은 서버에 세션을 저장하지 않는 stateless 구조였기 때문에, 토큰 자체가 아직 만료되지 않았다면 사용자가 로그아웃해도 같은 access token으로 API를 계속 호출할 수 있는 구간이 남아 있었습니다. 또한 refresh token 재발급 과정에서도 회전과 재사용 탐지를 강하게 적용하지 않으면 탈취 토큰을 조기에 끊기 어려웠습니다.

Cause

순수 stateless JWT만으로는 “지금 이 토큰을 더 이상 신뢰하지 않는다”는 서버 쪽 판단을 즉시 반영하기 어렵습니다. 특히 로그아웃, refresh rotation, 다중 세션 상한, 재사용 탐지처럼 토큰의 생애주기를 통제해야 하는 요구사항은 access/refresh 토큰 상태를 최소한의 서버 저장소와 함께 관리해야 해결할 수 있었습니다.

Solution

Refresh token은 Redis에 SHA-256 해시로 저장하고, access token은 JTI blacklist로 즉시 차단할 수 있게 분리했습니다. refresh 재발급에는 rotation과 재사용 탐지를 넣고, 사용자 5세션 상한과 관리자 단일 세션 정책까지 Redis key 구조 안에 포함해 “로그아웃 즉시 무효화”와 “세션 개수 통제”가 함께 동작하도록 인증 인프라를 재구성했습니다.

Result

로그아웃 후 같은 access token 재사용이 JwtFilter 단계에서 401로 차단되고, 회전 후 이전 refresh token 재사용도 재현 테스트에서 탐지되는 것을 확인했습니다. 단순 로그인 성공 여부를 넘어서, 세션 폐기·재사용 탐지·세션 상한까지 서버가 즉시 통제할 수 있는 JWT·세션 보안 기반을 갖추게 되었습니다.

소셜 로그인 토큰이 URL에 노출될 수 있는 보안 문제

PR #647
Problem

소셜 로그인 또는 소셜 회원가입 완료 후 access token과 signup 토큰을 redirect URL query parameter로 넘기면, 브라우저 히스토리와 서버 로그, Referer 헤더에 민감 정보가 남을 수 있었습니다. 기능은 동작하더라도 토큰 전달 경로 자체가 불필요하게 넓게 노출되는 상태였습니다.

Cause

SPA와 OAuth 콜백을 빠르게 연결하려다 보니, 백엔드가 프론트로 넘겨야 하는 인증 결과를 URL 파라미터로 실어 보내는 구조가 생겼습니다. 하지만 URL은 사용자가 직접 보지 않는 순간에도 브라우저 기록, 리버스 프록시 로그, 외부 이동 시 Referer 같은 다양한 경로로 복제될 수 있어 토큰 전달 채널로는 안전하지 않았습니다.

Solution

백엔드는 OAuth handoff 토큰을 URL 대신 Max-Age 60초, SameSite=Strict 조건의 단기 쿠키로 전달하고, 프론트의 OAuthCallbackPage 에서는 이 쿠키를 읽은 뒤 즉시 삭제하도록 바꿨습니다. 동시에 state 소비 순서도 원자적으로 조정해, 보안 경로를 줄이면서 중복 콜백 레이스 가능성까지 함께 낮췄습니다.

Result

소셜 로그인 이후 토큰이 브라우저 주소창과 히스토리에 남지 않게 되었고, 서버 로그나 Referer로 흘러갈 수 있는 노출 경로도 함께 차단했습니다. 즉 “동작하는 OAuth”에서 끝나지 않고, 토큰이 실제로 어떤 채널을 통해 전달되는지까지 통제하는 쪽으로 소셜 로그인 보안을 한 단계 올릴 수 있었습니다.

소셜 로그인 사용자 결제 시 404

PR #825
Problem

카카오 소셜 로그인 사용자가 결제를 시도할 때, 프론트엔드에서는 결제 페이지 진입 대신 404 오류가 떨어지는 문제가 QA에서 발견됐습니다. 사용자는 결제 준비 실패 원인을 알 수 없었고, 백엔드도 실제 원인과 다른 예외 경로를 타고 있었습니다.

Cause

일부 소셜 계정은 이메일이 비어 있었고, 이 값이 customer_email NOT NULL 제약에 걸리면서 DB 저장 단계에서 DataIntegrityViolationException이 발생했습니다. 그런데 이 예외가 원래는 동시 요청 충돌을 복구하기 위한 catch 블록으로 흘러들어가면서, 실제 제약 위반이 주문 없음처럼 처리되어 404로 바뀌고 있었습니다.

Solution

결제 INSERT 이전에 이메일 존재 여부를 명시적으로 검증해 제약 위반을 비즈니스 오류로 앞단에서 막았습니다. 동시에 프론트엔드 오류 분기와 메시지를 수정해, 결제 진입 실패가 404가 아니라 사용자 정보 보완이 필요한 케이스로 처리되도록 정리했습니다.

Result

이메일이 없는 소셜 계정으로 결제 경로를 다시 재현했을 때 더 이상 404로 오인되지 않았습니다. QA 단계에서 원인 확인에 쓰였던 DB 직접 조회 결과와 응답 흐름이 일치하게 바뀌었고, 사용자도 결제 실패 원인을 올바르게 안내받게 되었습니다.

LLM 응답 형식이 바뀌면 파싱 실패

PR #16
Problem

AI 회화 스크립트와 최종 피드백은 화면에서 바로 렌더링해야 했기 때문에 필드 구조가 조금만 달라져도 서비스 파싱이 깨졌습니다. 같은 기능을 다시 호출해도 응답 키 이름이나 배열 구조가 흔들리면 Spring DTO 역직렬화와 프론트 표시가 함께 실패했습니다.

Cause

LLM은 자연어 모델이라 프롬프트가 느슨하면 설명 문장, 불필요한 래핑 텍스트, 키 누락처럼 형식 편차를 만들 수 있습니다. 서비스 계층은 이 응답을 구조화된 JSON으로 가정하고 있었기 때문에, 모델 출력 규약이 약한 상태에서는 파싱 실패가 곧 사용자 기능 실패로 이어졌습니다.

Solution

프롬프트를 Role·Instruction·Context·Output Format으로 분리해 모델이 따라야 할 출력 계약을 더 명확히 했습니다. JSON Schema 수준으로 응답 구조를 제한하고, Spring DTO와 화면이 기대하는 필드를 기준으로 허용값을 좁혀 응답 형식이 흔들려도 서비스 계층이 안정적으로 받을 수 있게 정리했습니다.

Result

AI 응답이 사람에게 그럴듯한 문장인지보다 서비스가 끝까지 소비할 수 있는 구조인지에 초점을 맞추게 되었습니다. 스크립트 생성, 피드백 생성, 추천 후속 처리까지 같은 응답 계약을 공유하게 되면서 화면 전달 실패가 크게 줄었습니다.

외부 AI API 오류가 전체 코칭을 중단

PR #14
Problem

발음 평가와 최종 피드백은 Azure Speech와 OpenAI를 연달아 호출하는 구조였기 때문에, 한 단계라도 Timeout이나 파싱 오류가 나면 코칭 전체가 500으로 끊길 수 있었습니다. 사용자는 음성 입력에 성공해도 결과 화면을 받지 못하거나, 추천 단계 실패 때문에 앞선 피드백까지 잃는 상황이 생겼습니다.

Cause

초기 파이프라인은 외부 API 응답이 정상이라는 가정이 강했고, 실패 시 예외가 상위까지 전파되어 전체 흐름을 중단시키고 있었습니다. 특히 Azure 인식 실패, OpenAI 응답 JSON 파싱 오류, 추천 후속 처리 실패가 각각 독립적인 부분 실패가 아니라 전체 코칭 실패로 확대되고 있었습니다.

Solution

FastAPI 쪽에 발음 평가 엔드포인트 진단 로그와 예외 경로 로그를 추가하고, Azure 인식이 실패하면 STT fallback 텍스트와 기본 피드백으로 응답을 조립하도록 바꿨습니다. OpenAI 분석은 safe wrapper로 감싸 실패 시 빈 problemWords와 기본 메시지를 반환하게 했고, 추천 단계가 실패해도 앞서 생성된 최종 피드백은 보존하도록 Spring 쪽 흐름도 분리했습니다.

Result

외부 API 일부가 실패해도 사용자는 최소한의 피드백과 다음 단계 정보를 받을 수 있게 되었습니다. 로그에는 어느 단계에서 실패했는지 남기되, 응답은 가능한 범위의 학습 결과를 유지하도록 바뀌어 코칭 전체 중단 빈도를 낮췄습니다.

브라우저마다 녹음 포맷이 달라 발음 인식 실패

PR #13
Problem

사용자가 브라우저에서 녹음한 음성을 그대로 업로드하면 어떤 환경에서는 발음 평가가 되지만, 어떤 환경에서는 같은 기능이 바로 실패했습니다. 사용자는 마이크 권한과 녹음은 정상인데도 음성 인식이 안 되는 상황을 겪었습니다.

Cause

브라우저와 기기마다 MediaRecorder가 만드는 포맷과 codec이 달랐고, Azure Speech는 입력 포맷 편차에 민감했습니다. 서비스는 업로드된 음성을 동일한 형식으로 들어올 것처럼 가정하고 있었기 때문에, WebM 계열 파일이나 샘플링 차이가 그대로 인식 실패로 이어졌습니다.

Solution

FastAPI에서 업로드 음성을 FFmpeg 기반 16kHz mono WAV로 변환해 발음 평가 입력을 표준화했습니다. Azure 발음 평가가 실패할 때는 STT fallback과 기본 피드백을 반환하도록 해, 포맷 문제나 낮은 품질 입력에서도 코칭 흐름이 완전히 끊기지 않도록 보완했습니다.

Result

브라우저별 입력 편차가 서비스 장애로 바로 이어지지 않게 되었고, 서로 다른 녹음 환경에서도 최소한의 음성 평가 결과를 받을 수 있게 되었습니다. 음성이 완벽히 평가되지 않더라도 사용자 경험은 “실패”가 아니라 “제한된 결과”로 바뀌었습니다.

React–Spring–FastAPI 음성 연동 오류

PR #152
Problem

사용자 녹음이 React에서 정상적으로 끝나도 Spring을 거쳐 FastAPI로 전달되는 과정에서 업로드가 실패하거나, 응답은 왔는데 프론트에서 재생/표시가 끊기는 문제가 반복됐습니다. 세 서버가 모두 살아 있어도 경계면 계약이 조금만 어긋나면 코칭 체인이 쉽게 깨졌습니다.

Cause

프론트에서는 녹음 종료 직후 아직 최종 chunk 수집이 끝나기 전에 Blob을 넘기고 있었고, 일부 요청은 File 메타데이터가 불완전한 상태였습니다. Spring 쪽에서는 multipart 음성 파트를 일괄 application/octet-stream으로 바꿔 보내고 있어 원래의 content type 정보가 사라졌고, 그 결과 FastAPI가 음성 파일을 해석하는 데 필요한 힌트를 잃고 있었습니다.

Solution

React에서는 녹음 종료 후 최종 chunk를 모두 모은 뒤 File 객체를 완성해 업로드하도록 수정하고, 빈 녹음 데이터는 앞단에서 차단했습니다. Spring의 FastAPI 연동 클라이언트에서는 MultipartFile.getContentType()을 보존해 multipart 파트에 실어 보내도록 바꿨고, FastAPI 응답도 음성 평가 실패 시 기본 구조를 유지하도록 정리해 세 서버 간 계약을 안정화했습니다.

Result

녹음부터 업로드, 발음 평가, 피드백 응답, 재생까지의 흐름이 한 번에 이어지도록 안정성이 높아졌습니다. 실제 커밋에서는 프론트의 recorder 파일 확정, Spring의 content type 보존, FastAPI의 fallback 응답 보강이 순차적으로 반영되며 연동 오류를 줄였습니다.

채용 공고 목록 API에서 반복 집계와 풀스캔으로 응답이 느려진 문제

PR #951
Problem

채용 공고 목록 API는 목록 본문뿐 아니라 필터 옵션과 통계 수치까지 함께 내려주고 있었기 때문에, 같은 화면을 다시 열어도 매번 무거운 집계 쿼리가 반복됐습니다. 공고 수가 늘수록 목록 응답과 초기 진입 체감 속도가 함께 떨어졌습니다.

Cause

서비스 내부에서 Distinct 필터 옵션과 COUNT 통계를 요청마다 다시 계산하고 있었고, ACTIVE 공고 조회도 시간이 지나며 누적된 CLOSED 데이터까지 함께 스캔할 가능성이 큰 구조였습니다. 즉 병목은 단순 페이지 조회가 아니라, 반복 집계와 정렬 쿼리가 동시에 누적되는 데 있었습니다.

Solution

필터 옵션과 통계는 JobNoticeCacheService로 분리해 Redis 캐시를 적용하고, TTL을 용도별로 다르게 설정했습니다. 동시에 PostgreSQL에는 ACTIVE 공고만 대상으로 하는 부분 인덱스를 추가해, 목록 정렬 경로가 전체 테이블이 아니라 실제 조회 대상만 더 빠르게 타도록 바꿨습니다.

Result

요청 1건당 DB 쿼리 8~9개가 반복되던 구조에서, 필터 옵션·통계 경로를 캐시로 분리한 뒤 같은 조건의 2번째 요청부터 응답 시간이 눈에 띄게 줄어드는 것을 확인했습니다. 또한 EXPLAIN ANALYZE 기준으로 목록 정렬 경로가 Sequential Scan 대신 Index Scan 중심으로 전환되어, 공고 수가 커져도 반복 조회 비용을 더 안정적으로 제어할 수 있게 되었습니다.

100만 건 규모에서 deep pagination이 사실상 타임아웃 수준으로 느려진 문제

PR #1292
Problem

채용 공고 데이터가 100만 건 규모로 커지자, 뒤쪽 페이지로 갈수록 OFFSET 기반 목록 조회가 급격히 느려졌습니다. 특히 deep page에서는 실제 사용자 요청보다 DB가 앞의 수십만 행을 버리는 비용이 더 커져, 몇 초가 아니라 수십 초 단위까지 치솟는 구간이 생겼습니다.

Cause

기존 쿼리는 전체 행을 정렬한 뒤 OFFSET 만큼 건너뛰는 방식이라, 뒤 페이지일수록 버려지는 행의 비용이 그대로 누적됐습니다. 여기에 정렬 동률 시 tie-breaker가 없어 페이지 경계에서 중복·누락 가능성도 있었고, 1단계 조회와 실제 응답 조립 사이에 데이터가 바뀌면 결과 정합성도 흔들릴 수 있었습니다.

Solution

API 계약은 유지한 채, 먼저 페이지에 필요한 ID만 커버링 인덱스로 빠르게 뽑고 이후 해당 ID의 실제 행만 읽는 deferred join 2단계 방식으로 바꿨습니다. 또한 모든 정렬에 jobNoticeId tie-breaker를 추가하고, 2단계 조회에서도 동일 predicate를 다시 적용해 deep pagination에서도 순서와 결과 정합성이 흔들리지 않도록 보완했습니다.

Result

EXPLAIN ANALYZE 기준으로 OFFSET 50만 구간이 약 84초에서 약 137ms 수준으로 줄었고, 80만 구간도 타임아웃 대신 147ms대로 안정화했습니다. 프론트 API를 갈아엎지 않고도 deep page 병목만 정확히 제거해, 성능과 기존 UX를 함께 지킬 수 있었습니다.

구독 자동결제 배치에서 N+1 조회와 순차 외부 호출로 처리 시간이 길어진 문제

PR #1281
Problem

자동결제 배치는 대상 구독 수가 늘수록 한 건씩 필요한 정보를 다시 조회하고, Toss 결제 호출도 순차로 기다리는 구조였습니다. 배치량이 커지면 처리 완료까지 오래 걸릴 뿐 아니라, 스케줄러가 다음 작업을 미루게 될 위험도 커졌습니다.

Cause

구독, 요금제, 결제 프로필, 회원 정보, 멱등 결제 정보를 루프 안에서 개별 조회해 전형적인 N+1 패턴이 발생하고 있었습니다. 또한 외부 결제 API 호출을 순차 처리하고 있어, DB 병목과 네트워크 대기 시간이 동시에 전체 wall-clock 시간을 늘리고 있었습니다.

Solution

배치 시작 전에 필요한 연관 데이터를 IN 절 배치 조회로 선로딩해 RenewalContext로 묶고, 결제 생성도 선로딩 결과를 재사용하도록 구조를 바꿨습니다. 그 위에 고정 스레드풀 기반 bounded 병렬화를 적용하되, 멱등키·건당 타임아웃·취소 처리를 넣어 속도를 높이면서도 중복결제와 스케줄러 블로킹을 함께 막았습니다.

Result

건당 SQL 수는 10개에서 6개로 줄었고, 결제 진입 전 반복 조회는 3만 건 수준에서 사실상 한 자릿수 배치 조회로 축소됐습니다. 1만 건 기준 wall-clock은 64.4초에서 17.5초로, Toss 지연이 있는 환경에서도 235.6초에서 13.9초로 크게 단축하면서 결제 정확성은 그대로 유지했습니다.
